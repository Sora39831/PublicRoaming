## 4. 核心组件分析

### 4.1 XposedInit — 入口点

`XposedInit` 实现两个 Xposed 接口：

```kotlin
class XposedInit : IXposedHookLoadPackage, IXposedHookZygoteInit
```

**生命周期：**

```
1. initZygote(startupParam)
   └─ 保存 modulePath，创建 XModuleResources

2. handleLoadPackage(lpparam)
   ├─ 如果是自身进程 → hook isModuleActive 返回 true
   ├─ 如果不是 B 站进程 → 直接 return
   └─ hookBeforeMethod("callApplicationOnCreate")
      ├─ 主进程 (无冒号):
      │   ├─ 启动日志记录
      │   ├─ 检测当前地区 (中国/香港/台湾/海外)
      │   ├─ 实例化 BiliBiliPackage (加载 HookInfo)
      │   ├─ startHook() × 40+ (注册所有 Hook)
      │   └─ country = 异步获取地区信息
      │
      ├─ :web 进程:
      │   ├─ CustomThemeHook (Web 主题注入)
      │   ├─ WebViewHook
      │   ├─ ShareHook
      │   ├─ DialogBlurBackgroundHook
      │   └─ RewardAdHook
      │
      └─ :download 进程:
          └─ BangumiPlayUrlHook (仅播放地址替换)

3. lateInitHook → Activity.onResume
   └─ 遍历所有 hookers 调用 lateInitHook()
```

**错误处理策略：**
每个 Hook 的创建和初始化都被 try-catch 包裹，单个 Hook 失败不影响其他 Hook 运行，错误通过 Toast 和日志通知用户。

### 4.2 BiliBiliPackage — 反射元数据管理

这是整个模块最核心的类（2524行），承担以下职责：

#### 4.2.1 HookInfo 的读取与缓存

```kotlin
private fun readHookInfo(context: Context): Configs.HookInfo {
    val hookInfoFile = File(context.cacheDir, "hookinfo.pb")
    // 检查缓存是否仍然有效（版本匹配、模块匹配、B站版本匹配）
    if (info.lastUpdateTime >= lastUpdateTime
        && getVersionCode() == info.clientVersionCode
        && BuildConfig.VERSION_CODE == info.moduleVersionCode
        && info.generation >= getModuleGeneration(context)
        && info.biliAccounts.getAccessKey.orNull != null)
        return info  // 缓存命中
    return initHookInfo(context).also { /* 写入缓存 */ }
}
```

**缓存失效条件：**
- B 站客户端更新
- 模块自身更新
- 模块缓存版本号 (generation) 不匹配
- 关键符号（如 accessKey 获取方法）丢失

#### 4.2.2 反射符号的自动发现 (initHookInfo)

当缓存失效时，通过 C++ DEX 扫描引擎自动重新定位所有反射符号：

```kotlin
fun initHookInfo(context: Context) = hookInfo {
    System.loadLibrary("biliroaming")
    val dexHelper = DexHelper(classloader.findDexClassLoader())
    
    // 通过字符串特征定位方法
    // 例如: 在 OkHttp 中搜索 "Response{protocol=" 找到 Response 类
    val responseClass = dexHelper.findMethodUsingString(
        "Response{protocol=", false, -1, -1, null, -1,
        null, null, null, false
    ).asSequence().mapNotNull { dexHelper.decodeMethodIndex(it) }
     .firstOrNull { it.declaringClass?.name?.startsWith("okhttp3") == true }
     ?.declaringClass
}
```

**扫描策略：**
- **字符串定位**：通过硬编码字符串常量定位方法（如 `"toJSON error"` → fastjson）
- **调用图回溯**：`FindMethodInvoking` / `FindMethodInvoked` 追踪方法调用关系
- **字段访问追踪**：`FindMethodGettingField` / `FindMethodSettingField` 追踪字段读写
- **签名匹配**：按参数类型、返回类型、修饰符组合匹配方法

#### 4.2.3 单例模式与延迟加载

```kotlin
companion object {
    @JvmStatic lateinit var instance: BiliBiliPackage
    // Weak 引用确保 ClassLoader 可被 GC
    val bangumiApiResponseClass by Weak { mHookInfo.bangumiApiResponse from mClassLoader }
}
```

所有反射引用使用 `by Weak` 委托延迟加载，避免在类加载阶段触发大量反射操作。

### 4.3 BaseHook — Hook 抽象基类

```kotlin
abstract class BaseHook(val mClassLoader: ClassLoader) {
    abstract fun startHook()      // 主 Hook 逻辑
    open fun lateInitHook() {}    // 延迟初始化（Activity.onResume 后）
}
```

所有 Hook 继承此基类，遵循以下模式：
1. 构造时接收 ClassLoader
2. `startHook()` 中注册方法 Hook（替换/前/后）
3. 可选实现 `lateInitHook()` 做延迟初始化

### 4.4 HookInfo 缓存机制

**Protobuf 序列化**的 HookInfo 包含 100+ 个符号路径：

```
HookInfo {
    last_update_time          // 最后更新时间戳
    client_version_code       // 目标 B 站版本号

    bangumi_params            // 番剧参数类
    theme_helper              // 主题色数组
    drawer                    // 侧边栏相关
    ... 100+ 个字段
}
```

## 5. Native 层 (C++/JNI)

### 5.1 DexHelper — DEX 扫描引擎

Native 层基于 **LSPosed 的 DexBuilder** 项目，提供高性能 DEX 文件分析能力：

**核心能力：**

| JNI 方法 | 功能 |
|---------|------|
| `load()` | 映射进程所有 DEX 到内存 |
| `findMethodUsingString()` | 在常量池中搜索字符串，定位引用方法 |
| `findMethodInvoking()` | 查找调用了指定方法的方法 |
| `findMethodInvoked()` | 查找被指定方法调用的方法 |
| `findMethodGettingField()` | 查找读取指定字段的方法 |
| `findMethodSettingField()` | 查找写入指定字段的方法 |
| `findField()` | 按类型查找字段 |
| `decodeMethodIndex()` | 解码方法索引为 Java Method |
| `decodeFieldIndex()` | 解码字段索引为 Java Field |
| `decodeClassIndex()` | 解码类索引为 Java Class |
| `encodeMethodIndex()` | 编码 Java Method 为索引 |
| `createFullCache()` | 创建完整缓存加速后续查询 |

**DEX 内存映射：**

```cpp
struct MemMap {
    uint8_t *addr_;  // mmap 映射地址
    size_t len_;     // DEX 文件大小
    // 支持 ZIP 解压（Compact DEX 回退到原始文件）
};
```

**处理 Compact DEX：**
当检测到 Compact DEX 格式时，从原始 APK 文件中通过 ZIP 解压提取 `classes.dex`, `classes2.dex`, ... 等。

### 5.2 JNI 桥接方法

`DexHelper.java` 声明 native 方法，通过 JNI 调用 C++ 实现：

    public native long[] findMethodUsingString(String str, boolean matchPrefix, ...);

## 6. Hook 功能分类

### 6.1 番剧区域限制解除

| Hook | 功能 | 复杂度 |
|------|------|--------|
| **BangumiSeasonHook** | 番剧季信息替换，向服务器请求港澳台/泰国服数据，修改 area_limit、rights 等字段 | ★★★★★ (1720行) |
| **BangumiPlayUrlHook** | 番剧播放地址替换，将大陆 CDN 替换为港澳台 CDN，或直接使用解析服务器 | ★★★★☆ (1128行) |

**工作流程：**
```
用户播放番剧
    ↓
BiliRoamingApi.getSeason() → 从 BiliPlus/Bili 国际服获取区域可用番剧信息
    ↓
修改 response: 设置 area_limit=0, allow_download=1, rights 解锁
    ↓
BiliRoamingApi 获取可用播放地址
    ↓
替换 CDN 节点（大陆→港澳台），或直接标记为可用
    ↓
播放器获取到区域解锁的播放流
```

### 6.2 广告屏蔽

|------|------|
| **SplashHook** | 屏蔽/替换开屏广告，支持自定义启动图 |

| **StoryPlayerAdHook** | Story 播放器广告移除 |

| **PegasusHook** | 首页信息流卡片过滤（含广告卡片） |
| **ProtoBufHook** | Protobuf 通用广告字段清理 |

### 6.3 视频画质增强

| Hook | 功能 |

## 12. 关键设计模式

### 12.1 防御式编程

```kotlin
private fun startHook(hookerCreater: () -> BaseHook) {
    try {
        val hooker = hookerCreater()
        hookers.add(hooker)
        hooker.startHook()
    } catch (e: Throwable) {
        Log.e(e)
        Log.toast("出现错误\n${e.message}\n部分功能可能失效。")
    }
}
```

每个 Hook 独立 try-catch，确保单点故障不影响全局。

### 12.2 缓存优先的符号解析

```
首次启动 / B站版本更新:
    initHookInfo() → C++ DEX 扫描 → 符号定位 → Protobuf 序列化 → 写入 hookinfo.pb

后续启动:
    readHookInfo() → 版本校验 → 直接反序列化 → 使用缓存符号
```

### 12.3 弱引用委托

```kotlin
val bangumiApiResponseClass by Weak { mHookInfo.bangumiApiResponse from mClassLoader }
```

使用 `Weak` 委托避免 ClassLoader 泄漏，同时实现延迟加载。

### 12.4 协程异步

```kotlin
country = MainScope().future(Dispatchers.IO) {
    fetchJson(Constant.zoneUrl)?.optJSONObject("data")?.optInt("country_code", 0)
        .let { when(it) { 86 -> "cn"; ... } }
}
```

地区检测和番剧信息获取使用协程异步执行，避免阻塞主线程。

### 12.5 JSON 多后端兼容

通过 `JsonHelper` 接口抽象，支持从 Fastjson 到 Gson 到 kotlinx.serialization 的迁移，通过 `isGson` / `isSerializable` 等标志动态选择解析路径。

### E. Hook 实现细节补充

基于全部 58 个 Hook 源码的详细分析，以下补充关键实现机制：

#### 协议覆盖
模块拦截了 B 站客户端的全部网络协议层：
- **Retrofit HTTP (JSON 响应)** → `JsonHook` (fastjson) / `GsonHook` / `KotlinxJsonHook`
- **Moss RPC (Protobuf 响应)** → `ProtoBufHook` / `BangumiSeasonHook` / `PegasusHook`
- **PlayURL v1/v2** → `BangumiPlayUrlHook` (PlayURLMoss + PlayerMoss.playViewUnite)
- **弹幕 (DmViewMoss)** → `DanmakuHook` / `ProtoBufHook`

#### JSON 多后端兼容
B 站在不同版本切换了 JSON 框架，模块并行维护三套拦截器：
1. **Fastjson** (旧版) → `JsonHook` + `FastjsonHelper`
2. **Gson** (中期) → `GsonHook` + `GsonHelper` + `GsonProcessor`
3. **kotlinx.serialization** (新版) → `KotlinxJsonHook` + `KotlinxProcessor`

#### Protobuf DSL `copy{}` 模式
Kotlin Protobuf lite 生成的代码支持不可变对象的 `copy{}` 构建器语法，模块大量使用此语法进行类型安全的字段修改：
```kotlin
// ProtoBufHook 中的典型用法
 PlayViewReply.copy {
     playConf = playAbilityConfCopy {
         backgroundPlayDisable = false
         playbackRateDisable = false
     }
 }
```

#### Moss Response Handler Proxy
对于使用 JDK `Proxy` 动态代理的 Moss 响应处理路径：
```kotlin
mossResponseHandlerProxy { param ->
    // 拦截 ResponseHandler.onNext，在反序列化前/后修改数据
}
```

#### 版本差异化适配
`BiliBiliPackage` 中维护了多个版本标志（`useNewMossFunc`、`hasUnitedVideoActivity`），各 Hook 根据当前 B 站版本选择不同的 Hook 目标和策略。`SettingHook` 中甚至有针对 8.97.0 版本的专用 Adapter 注入路径。
